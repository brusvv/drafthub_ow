# Draft Hub V2 — экран «Карты»

> Проектирование следующего полноценного Workspace-экрана V2 после Heroes.
> Документ создан с чистого листа: существующие mockup-файлы не использовались
> как основа. Все решения ниже выводятся из Frozen v1.0 `DESIGN_PHILOSOPHY.md`
> и `DESIGN_SYSTEM.md`; сами Frozen-документы в рамках задачи не меняются.
>
> Статус Heroes: завершён как первый экран V2 — есть спецификация,
> интерактивный mockup и Post Validation. Maps не наследует Heroes-макет
> автоматически; совпадающие решения здесь повторены только там, где они
> независимо следуют из Design System.

---

## 1. Цель экрана

Экран «Карты» — рабочий каталог карт Overwatch для подготовки к драфту и
оценки решений по map pool. Он отвечает на четыре вопроса:

1. Какие карты есть в текущем scope?
2. Как быстро найти карту по имени или типу режима?
3. Какие карты требуют внимания перед матчем: не оценены, слабые для команды,
   имеют мало данных по силе героев?
4. Стоит ли открыть detail-профиль карты для решения по героям, сторонам и
   заметкам?

Экран не является generic analytics dashboard: агрегаты появляются только там,
где они ведут к следующему действию — например «3 карты без оценки» рядом с
фильтром/переходом к этим картам.

---

## 2. Структура экрана

### 2.1 Shell

Используется существующая навигационная оболочка:

- identity-полоса с командой/scope/пользователем;
- nav-рельс с активным пунктом «Карты»;
- primary/secondary-группы навигации без изменений.

Shell не содержит map-specific действий. Действия экрана живут в Workspace,
потому что в Design System нет подтверждённого слоя Global Actions.

### 2.2 Workspace

Workspace состоит из пяти зон:

1. **Page header** — заголовок «Карты» и короткая сводка текущего scope.
2. **Action summary** — компактная сводка проблем, которые меняют следующее
   действие пользователя.
3. **Toolbar** — search, toggle-группа типов карт, sort/select.
4. **Map groups** — группировка карт по типу режима.
5. **Compact map grid** — карточки карт.

### 2.3 Overlay / Detail

Detail карты открывается из compact-карточки и показывает данные, которые нельзя
размещать в сетке без потери плотности:

- лучшие/слабые герои на карте;
- заметки по сторонам атака/оборона, если применимо;
- team/personal overrides;
- связанные tier/counter данные.

Как и в Heroes, это локально использует существующий Detail popup / Overlay,
но не объявляет новый системный компонент.

---

## 3. Компоновка

### 3.1 Общий layout

- Desktop-first Workspace внутри Shell.
- Padding Workspace: значения из существующей spacing-шкалы.
- Контент скроллится внутри Workspace, Shell остаётся стабильным.
- Карты отображаются compact-сеткой, а не таблицей.

### 3.2 Header

Header содержит:

- H1 `Карты`;
- secondary text: активный scope и количество карт;
- без Primary-кнопки, если пользователь только просматривает каталог.

Primary появляется только при реальном обязательном действии, например в empty
state «Импортировать карты» или в error state «Повторить загрузку» не как
primary screen action, а как действие состояния.

### 3.3 Action summary

Сводка не является набором декоративных KPI. Разрешены только карточки, которые
меняют следующее действие:

- `Без оценки` — количество карт без team/personal оценки;
- `Слабые карты` — количество карт ниже выбранного порога;
- `Мало данных` — карты без достаточной привязки hero strength.

Каждый summary item кликабелен как фильтр или приводит к соответствующей группе.
Если все показатели нулевые, summary сворачивается в спокойный статус «Карты
готовы к драфту».

### 3.4 Toolbar

Toolbar содержит:

- Search input по имени карты;
- toggle-группу типов карт: `Все / Control / Escort / Hybrid / Push / Flashpoint / Clash`;
- Select `Сортировка`: `по типу`, `по оценке`, `по имени`, `по обновлению`;
- съёмные фильтр-чипы только для дополнительных multi-select фильтров, если они
  появятся позже; в базовой версии не нужны.

Тип карты — взаимоисключающий фильтр в этом экране, поэтому используется toggle,
а не removable chips.

### 3.5 Content hierarchy

Порядок внутри списка:

1. Type section header: `Control`, `Escort`, `Hybrid`, ...
2. Compact map grid.
3. Card-level badges/status.

В отличие от Heroes, второй уровень `тип → подтип` не нужен: тип карты уже
является основной доменной группой, а дополнительная вложенность ухудшит
сканирование без доказанной пользы.

## 4. Используемые компоненты

| Область | Компонент Design System | Использование |
| --- | --- | --- |
| Shell | Navigation shell | Active section: Maps |
| Header | H1 + Body Small | Название раздела и meta текущего scope |
| Action summary | Stat Card | Только метрики, которые ведут к действию |
| Search | Search input | Поиск карты по имени |
| Map type | Toggle-группа | Взаимоисключающий фильтр типов |
| Sort | Select | Порядок отображения сетки |
| Map item | Compact card | Баннер/скриншот, имя, тип, 1-2 ключевых статуса |
| Status | Status badge | `Без оценки`, `Мало данных`, `Слабая` |
| Loading | Skeleton | Повторяет форму summary + card grid |
| Empty | Empty state | Причина + действие |
| Error | Alert | Безопасный текст + повторить загрузку |
| Detail | Detail popup / Overlay | Полный профиль карты |

Новых компонентов не добавляется.

## 5. Состояния

### 5.1 Default

Показывает:

- header;
- action summary, если есть actionable проблемы;
- toolbar;
- группы карт по типу;
- compact-карточки карт.

Compact-карточка карты содержит:

- баннер/скриншот или placeholder;
- имя карты;
- type badge;
- один основной статус: `Без оценки`, `Мало данных`, `Слабая`, либо спокойное
  отсутствие status badge, если карта заполнена и не требует внимания;
- optional score/tier, если он является уже существующей доменной метрикой scope.

Карточка не показывает полный список сильных героев: это detail-уровень.

### 5.2 Loading

Loading состоит из двух skeleton-зон:

1. skeleton summary row;
2. skeleton grid карт.

Не используется spinner по центру. Skeleton повторяет форму будущего контента и
сохраняет положение toolbar/header, чтобы пользователь не терял контекст.

### 5.3 Empty — каталог пуст

Если в текущем scope нет карт:

- заголовок: `Карты не добавлены`;
- текст: объясняет текущий scope;
- действие: `Импортировать карты` или `Перейти в Global`, в зависимости от прав.

Если у пользователя нет прав импортировать карты, действие становится Secondary
`Показать Global-карты`, а текст объясняет ограничение прав.

### 5.4 Empty — фильтр без результатов

Если каталог есть, но search/type filter ничего не нашли:

- заголовок: `Ничего не найдено`;
- текст: явно называет активный запрос/тип;
- действие: `Сбросить фильтры`.

Это отдельное empty-состояние, не смешивается с пустым каталогом.

### 5.5 Error

Ошибка загрузки карт показывается Alert внутри Workspace:

- безопасный заголовок: `Не удалось загрузить карты`;
- безопасный текст без сырого exception/message;
- действие `Повторить загрузку`;
- вторичная подсказка: локальные фильтры сохраняются.

Toast не подходит, потому что без карт раздел не выполняет задачу.

### 5.6 Detail popup

Открывается по клику/Enter на карточке карты.

Содержит:

- крупный баннер карты;
- имя и type badge;
- score/tier текущего scope;
- блок `Сильные герои`;
- блок `Слабые герои / риски`;
- блок `Заметки по сторонам`, если карта поддерживает разные стороны;
- действия Secondary/Ghost: `Редактировать оценку`, `Закрыть`.

Danger-действия в detail не показываются без отдельного подтверждения.

## 6. Пользовательские сценарии

### Сценарий A — аналитик готовит map pool

1. Открывает «Карты».
2. Смотрит Action summary.
3. Нажимает `Без оценки`.
4. Видит только карты, где не хватает данных.
5. Открывает detail карты.
6. Переходит к редактированию оценки или закрывает detail после просмотра.

### Сценарий B — тренер быстро ищет конкретную карту

1. Вводит имя карты в Search.
2. Сетка фильтруется без Enter.
3. Открывает карту.
4. Сверяет сильных/слабых героев и возвращается к списку.

### Сценарий C — подготовка к режиму карты

1. Нажимает toggle `Control` или другой тип.
2. Видит только карты выбранного типа.
3. Сортирует по оценке.
4. Открывает слабые карты первыми.

### Сценарий D — данные не загрузились

1. Видит Alert, а не пустой каталог.
2. Нажимает `Повторить загрузку`.
3. При успехе возвращается к default; при повторной ошибке Alert остаётся.

---

## 7. Взаимодействия

- Search фильтрует при вводе с debounce не длиннее базовой motion-длительности.
- Type toggle взаимоисключающий; active-состояние не кодируется только цветом.
- Sort select не меняет выбранные фильтры.
- Summary item применяет фильтр и визуально связывает причину со списком.
- Card hover использует border/surface tokens, не тени.
- Card focus-visible обязателен.
- Detail popup закрывается Escape, close button и click outside.
- Error retry не сбрасывает search/type filters.

---

## 8. Обоснование ключевых решений

| Решение | Обоснование |
| --- | --- |
| Compact grid | Карты — доменная сущность; таблицы зарезервированы для admin/служебных данных. |
| Action summary только для проблем | Метрика попадает на экран, только если меняет следующее действие. |
| Search | Карты входят в плотные списки, где быстрый поиск нужен для конкретной сущности. |
| Toggle типов | Тип карты в базовом сценарии взаимоисключающий фильтр. |
| Отдельные empty states | Пустой каталог и пустой фильтр требуют разных действий. |
| Alert для error | Ошибка загрузки должна быть персистентной и actionable. |
| Detail по клику | Полные hero/map-strength данные не помещаются в compact card без нарушения плотности. |
| Без второго уровня группировки | Для Maps нет независимого доказательства пользы `тип → подтип`; лишняя вложенность ухудшит сканирование. |

# Post Validation

## Какие правила Design System использовались

- Shell → Workspace → Overlay.
- Navigation shell без новых глобальных действий.
- Neutrals/Semantic/domain colors без новых токенов.
- Sans для читаемых названий, Mono для meta/status labels.
- Spacing/radius/motion только из фиксированных шкал.
- Search input.
- Toggle-группа.
- Select.
- Compact/Detail card density.
- Stat Card только для actionable aggregate metrics.
- Status badges.
- Loading / Empty / Error обязательная тройка.
- Alert вместо Toast для ошибки загрузки.
- Focus-visible и redundant state indicators.

## Какие компоненты оказались достаточными

- Navigation shell.
- Search input.
- Toggle-группа.
- Select.
- Compact card.
- Stat Card.
- Status badge.
- Skeleton.
- Empty state.
- Alert.
- Detail popup / Overlay.
- Secondary/Ghost/Danger button taxonomy.

## Какие локальные решения пришлось принять

1. **Action summary для Maps.** Design System разрешает Stat Card точечно, но не
   перечисляет конкретные map-метрики. Для Maps выбраны только actionable
   агрегаты: `Без оценки`, `Слабые карты`, `Мало данных`.

2. **Type toggle как single-select.** Тип карты можно было бы трактовать как
   multi-select removable chips, но базовый сценарий «посмотреть один режим»
   проще и быстрее как взаимоисключающий toggle. Это локальное решение экрана,
   не общее правило для всех фильтров карт.

3. **Нет второго уровня группировки.** Heroes обнаружил вопрос content hierarchy;
   Maps сознательно не повторяет `группа → подгруппа`, потому что из данных
   экрана это не выводится независимо.

4. **Detail popup для карты.** Как и у Heroes, полный профиль карты временно
   укладывается в существующий Detail popup / Overlay. Вопрос отдельной
   информационной панели остаётся локально отмеченным, но одного повторения на
   Maps пока недостаточно, чтобы менять Frozen Design System в этой задаче.

## Какие проблемы действительно обнаружены

- Нужен будущий критерий, когда Stat Card является actionable, а когда превращается
  в декоративный KPI. На Maps правило применимо, но требует дисциплины контента.
- Detail popup / informational panel повторно появляется как вопрос после Heroes:
  если то же подтвердится на следующем экране сущностей, появится более сильное
  основание формализовать отдельный detail-container.
- Права пользователя в empty state требуют точного продуктового решения:
  `Импортировать карты` vs `Показать Global-карты` зависит от permission model,
  не от визуальной системы.

## Есть ли реальные основания менять Design System

Пока **изменений Design System не требуется**. Maps смог быть спроектирован без
новых компонентов и токенов. Есть повторяющийся сигнал по informational detail
container после Heroes + Maps, но правило пользователя запрещает менять систему
по ходу проектирования одного экрана. Это нужно проверить ещё на следующем
экранe сущностей; если проблема повторится, тогда Post Validation следующего
экрана сможет стать основанием для отдельного изменения Design System.
