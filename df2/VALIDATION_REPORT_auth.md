# Validation Report — экран «Auth» (20.07, продолжение)

> Выбран специально не как очередной «обычный» экран, а чтобы проверить
> гипотезу о `Shell` из `DESIGN_SYSTEM.md` §0: является ли отсутствие
> Shell у Share Token изолированным случаем или частью общего паттерна.
> Артефакт: `auth-screen-mockup.html`, 4 состояния (обычная работа/вход/
> регистрация/нет команд), проверено рендером через headless Chrome
> (скриншоты всех 4 состояний), не только чтением кода.

---

## 1. Auth в V1 — что это структурно

Проверено построчно в реальном коде, не по памяти:

- `build.sh` (`HTML_PARTS`) подключает `auth.html` и `main-app.html` как
  два **соседних** партиала, оба всегда присутствуют в итоговом
  `index.html`. `main-app.html` оборачивает Shell+Workspace в
  `<div id="app" style="display:none">` — скрыт по умолчанию.
  `auth.html` — просто `<div id="authScreen">`, контент рендерится
  динамически.
- `auth/ui.js` (`renderAuthUI()`, строки 9-22) переключает видимость
  между `#authScreen`/`#app` через `.style.display` — то есть это
  переключение между **двумя постоянными контейнерами одного уровня**,
  не показ/скрытие чего-то ВНУТРИ Shell.
- `_renderLoginForm()`/`_renderRegisterForm()`/`_renderNoTeams()` — все
  три возвращают один и тот же паттерн: `.auth-screen` → `.auth-logo` +
  `.auth-card`. Ни намёка на identity-bar/rail — это самостоятельный,
  bare-card layout, не урезанная версия Shell.

**Вывод:** Auth в V1 — не состояние приложения (не "Workspace без
Shell"), а второй, полностью независимый top-level контейнер, живущий
рядом с `#app` в том же документе.

## 2. Присутствует ли Shell — и от чего это зависит

Не присутствует ни в одном из auth-состояний, включая то, где сессия
РЕАЛЬНО есть:

- `login`/`register` — сессии нет, ожидаемо.
- `no-teams` (`_renderNoTeams()`) — сессия уже есть (доступна кнопка
  «Выйти», `signOut()`), но команд у пользователя — 0. Shell всё равно
  отсутствует.

**Это меняет формулировку критерия.** Не «есть сессия → Shell есть,
нет сессии → Shell нет» — а точнее: **Shell требует активный командный
контекст, не просто аутентификацию.** Пользователь может быть залогинен
и всё равно не увидеть Shell, если ему ещё нечего в нём показывать
(нет команды → нет данных → нечего связывать с identity-bar/rail).

## 3. Совпадает ли механизм с Share Token — нет, хотя причина похожая

Сверено построчно с `render-tier-share-public.js` (реализация Share
Token из прошлого разбора). Причина отсутствия Shell у обоих —
одного рода (нет активного командного/сессионного контекста, которым
Shell мог бы что-то показать) — **но реализованы двумя разными,
независимыми механизмами**, не одним и тем же паттерном дважды:

| | Auth | Share Token |
|---|---|---|
| Механизм | Toggle видимости между двумя **постоянными** sibling-контейнерами (`#authScreen`/`#app`), оба всегда в DOM | `document.body.innerHTML` — **одноразовая полная замена** всего body; `#app`/`#authScreen` перестают существовать в DOM вообще |
| Токены/классы | Использует общий `<style>` документа как есть — те же CSS custom properties, что и Shell/Workspace | Success-путь внедряет **свой** `<style>`-блок + переиспользует `.app` **класс** (не `#app` id) для внешнего контейнера и часть компонентных классов (`.tier-hero-pill`, `.f-btn`) — частичное, избирательное переиспользование. Error-путь — вообще мимо системы, хардкод `#0d0d0f`/`Inter,sans-serif` инлайном, без единой CSS-переменной |
| Место в SPA-жизненном цикле | Часть обычного цикла — рендерится/перерисовывается через `renderAuthUI()` как любой другой view | Перехватывает **до** обычного запуска приложения (`session.js`, вызывается из `initSession()` раньше, чем решается auth/app), это не «ещё один view», это отдельный маршрут |

**Вывод:** гипотеза «Shell — контекстно-зависимый, не обязательный
слой» — подтверждается на обоих примерах, но **не одним и тем же
способом**. Это два независимых архитектурных решения с общей причиной,
не один паттерн, montrise дважды.

## 4. Открытый вопрос — не решаю здесь

`DESIGN_SYSTEM.md` §0.3/§0.4 описывает Shell/Workspace/Overlay как три
слоя, из которых состоит экран. И Auth, и Share Token показывают, что
этой модели не хватает ещё одной оси — **не слой конкретного экрана, а
режим рендеринга самого приложения**: «внутри SPA-цикла (Shell/Workspace/
Overlay имеют смысл обсуждать)» против «отдельный маршрут в обход
SPA-цикла (эти три слоя неприменимы по определению, не потому что один
из них "выключен")». Auth — пример первого (toggle внутри цикла), Share
Token — пример второго (полная замена body, до/вне цикла).

Не предлагаю формулировку этой оси сейчас — по договорённости
(`DESIGN_SYSTEM.md` не трогаем до Meta Validation Report). Фиксирую как
находку, которая напрямую относится к §0, для того захода.
