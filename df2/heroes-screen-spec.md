# Draft Hub V2 — экран «Герои»

> Проектирование первого полноценного Workspace-экрана V2 на основе Frozen v1.0
> документов `DESIGN_PHILOSOPHY.md` и `DESIGN_SYSTEM.md`. Эти документы не
> меняются в рамках задачи. Если системе чего-то не хватило, это фиксируется
> только в Post Validation ниже.
>
> Интерактивный артефакт: `heroes-screen-mockup.html`.

---

## 1. Цель экрана

Экран «Герои» — рабочий каталог героев для аналитика/тренера. Его задача не
«показать все данные о герое сразу», а быстро ответить на три вопроса:

1. Какие герои доступны в текущем scope?
2. Как быстро найти героя по роли/имени?
3. Стоит ли открыть detail-профиль конкретного героя для решения по картам,
   контрпикам или синергиям?

Это следует из философии продукта: плотные списки — default для десятков
сущностей, а подробности раскрываются прогрессивно, только когда пользователь
выбрал конкретную сущность.

---

## 2. Структура экрана

Экран использует три слоя приложения из Design System:

### 2.1 Shell

- Верхняя identity-полоса: продукт, scope (`Global / Team / Personal`), команда
  и пользователь.
- Левый nav-рельс: primary-разделы сверху, secondary-разделы ниже разделителя.
- Активный раздел — «Герои».

Shell не переизобретается для Heroes: берётся существующий компонент
навигационной оболочки, потому что задача экрана — Workspace, а не новая
навигационная модель.

### 2.2 Workspace

Workspace состоит из четырёх зон:

1. **Toolbar раздела** — фильтр по роли и поиск.
2. **Список ролей** — группы `Tank`, `Damage`, `Support`.
3. **Подгруппы внутри роли** — подклассы/категории героев.
4. **Compact-сетка героев** — карточки героев.

### 2.3 Overlay

Detail-профиль героя открывается поверх Shell и Workspace. Это локальное решение
для текущего экрана: `DESIGN_SYSTEM.md` уже допускает `Detail (попап/панель)` у
карточек сущностей, но граница между информационным detail popup и модалкой как
«блокирующим решением» пока не до конца формализована. Поэтому решение не
распространяется автоматически на остальные экраны и фиксируется в Post
Validation.

---

## 3. Компоновка

### 3.1 Desktop frame

- Основной сценарий — desktop, не mobile-first.
- Shell занимает фиксированные зоны: identity-полоса `56px`, nav-рельс `220px`.
- Workspace скроллится независимо от Shell.
- Внутренний padding Workspace: `24px 28px` из существующей spacing-шкалы.

### 3.2 Toolbar

Toolbar расположен сверху Workspace и не является отдельным глобальным action
layer.

Состав:

- слева: toggle-группа ролей `Все / Tank / Damage / Support`;
- справа: Search input с мгновенной фильтрацией.

Обоснование:

- роль — взаимоисключающий фильтр, значит это toggle-группа, а не съёмные чипы;
- поиск обязателен для плотных списков героев/карт/игроков;
- primary-кнопка не нужна: на этом экране нет одного обязательного следующего
  действия уровня «создать/начать».

### 3.3 Content hierarchy

Порядок контента:

1. Role section header (`Tank`, `Damage`, `Support`) — читаемый Sans-заголовок.
2. Subrole label (`Initiator`, `Bruiser`, `Flanker` и т.д.) — Mono Section Title.
3. Compact hero grid — карточки одной плотности.

Локальное решение: двухуровневая группировка контента `Role → Subrole` не описана
как отдельный паттерн в Design System. В макете она решена через уже имеющиеся
типографические уровни, без добавления нового компонента.

---

## 4. Используемые компоненты

| Область | Компонент из Design System | Использование |
| --- | --- | --- |
| Shell | Навигационная оболочка | Identity + nav rail, Heroes active |
| Role filter | Toggle-группа | Взаимоисключающий фильтр ролей |
| Search | Search input | Мгновенная фильтрация по имени героя |
| Hero item | Compact card | Портрет/плейсхолдер, имя, роль-акцент, 1 слот метрик |
| Tier/role/status | Badge semantics | Role colors, banned status with text `БАН` |
| Detail | Detail popup / Overlay | Полный профиль героя |
| Loading | Skeleton | Повторяет форму сетки карточек |
| Empty | Empty state | Причина + действие сброса фильтра |
| Error | Alert | Персистентная ошибка + повторить загрузку |

Новых компонентов не вводится. Используются существующие компоненты и паттерны;
недостающие вопросы вынесены в Post Validation.

## 5. Состояния

### 5.1 Default

Показывает:

- toolbar;
- все роли;
- внутри каждой роли — subrole-группы;
- compact-карточки героев.

Карточка героя содержит:

- портрет/плейсхолдер;
- цветовую полосу роли;
- имя героя;
- до трёх мини-элементов в одном смысловом слоте «топ-контрпики»;
- status badge `БАН`, если герой недоступен/забанен.

Состояние не кодируется только цветом: role color дополняет текстовую группировку
и позицию в role section; ban имеет текстовый бейдж.

### 5.2 Loading

- Toolbar остаётся видимым, чтобы экран не прыгал структурно.
- Content заменяется skeleton-сеткой.
- Skeleton повторяет форму compact-карточек, а не показывает spinner в центре.
- Role headers не показываются до данных, потому что до ответа каталога нельзя
  гарантировать распределение героев по ролям.

### 5.3 Empty

Пустое состояние возникает после фильтра/поиска.

Показывает:

- что именно не найдено: роль и/или запрос;
- конкретное действие: `Сбросить фильтр`;
- без технических деталей.

Если активен только текстовый поиск при `Все`, действие формулируется как
изменение запроса, без лишней кнопки роли.

### 5.4 Error

Ошибка загрузки каталога/портретов/данных команды показывается как persistent
Alert внутри Workspace.

Показывает:

- короткий заголовок ошибки;
- безопасный текст без сырого технического сообщения;
- действие `Повторить загрузку`;
- вторичную подсказку, что сохранённые локальные фильтры не сброшены.

Toast не используется, потому что пользователь может пропустить исчезающее
сообщение, а раздел без данных не выполняет задачу.

### 5.5 Detail popup

Открывается кликом/Enter по compact-карточке.

Показывает:

- портрет;
- имя;
- роль + подкласс;
- priority-индикатор с точками и числом `8/10`;
- силу героя на картах списком;
- потенциально: counters/synergy/notes, если данные есть.

Закрывается:

- кнопкой закрытия;
- Escape;
- кликом по overlay-фону.

## 6. Пользовательские сценарии

### Сценарий A — аналитик ищет героя до матча

1. Открывает «Герои».
2. Вводит имя в Search.
3. Видит мгновенно отфильтрованную сетку.
4. Открывает detail конкретного героя.
5. Сверяет силу на картах и контрпики.
6. Закрывает detail и возвращается в ту же позицию сетки.

### Сценарий B — тренер быстро смотрит роль

1. Нажимает `Tank` / `Damage` / `Support`.
2. Сетка перестраивается только на выбранную роль.
3. Subrole labels помогают быстро просканировать группу.
4. Бейдж `БАН` сразу показывает недоступных героев без reliance только на цвет.

### Сценарий C — фильтр ничего не нашёл

1. Пользователь выбирает роль и вводит запрос.
2. Совпадений нет.
3. Empty state объясняет причину.
4. Пользователь нажимает `Сбросить фильтр` и возвращается к полной сетке.

### Сценарий D — данные не загрузились

1. Экран не показывает пустую сетку как будто данных нет.
2. Alert объясняет, что каталог не загрузился.
3. Пользователь нажимает `Повторить загрузку`.
4. После успеха возвращается default/loading flow.

---

## 7. Взаимодействия

- Role toggle меняет выбранную роль без перезагрузки страницы.
- Search фильтрует при вводе, debounce не длиннее базовой motion-длительности.
- Hero card является `button`-подобным интерактивным элементом и имеет
  `:focus-visible`.
- Hover у карточки меняет border/elevation только через существующие surface и
  border tokens.
- Banned hero остаётся видимым, но получает opacity/grayscale и текстовый бейдж.
- Detail popup сохраняет контекст Workspace под overlay и не меняет активный
  раздел в Shell.

---

## 8. Обоснование ключевых решений

| Решение | Основание |
| --- | --- |
| Compact-сетка вместо таблицы | Герои — доменная сущность, таблицы оставлены для admin/служебных данных. |
| Детали по клику | Прогрессивное раскрытие: карточка решает «открывать/не открывать», полный профиль не размазывается по сетке. |
| Search обязателен | Герои — плотный список из десятков сущностей, поиск нужен для быстрого доступа. |
| Toggle roles | Роль взаимоисключающая, значит подходит toggle-группа. |
| Empty с кнопкой | Каждый экран отвечает на «что дальше», пустое состояние требует действия. |
| Alert для ошибки | Ошибка загрузки не должна исчезать как toast и не должна быть сырым текстом исключения. |
| Без новых цветов | Роли/тиры/semantic colors уже покрывают нужные смыслы. |
| Без новой навигации | Shell уже решён; Heroes проектирует Workspace, а не новый nav-паттерн. |

# Post Validation

## Какие правила Design System использовались

- Анатомия приложения: Shell → Workspace → Overlay.
- Цветовые токены Neutrals/Semantic/Role/Tier.
- Типографика Sans vs Mono.
- Spacing/radius/motion из фиксированных шкал.
- Кнопки: Secondary для сброса/повтора, без лишней Primary.
- Search input.
- Toggle-группа ролей.
- Compact/Detail карточки сущности.
- Badge semantics для роли/тира/status.
- Alert / Empty / Loading правила для data-dependent section.
- `:focus-visible` и redundant state indicators.

## Какие компоненты оказались достаточными

- Навигационная оболочка.
- Toggle-группа.
- Search.
- Compact card.
- Detail popup/Overlay как временно достаточный контейнер для профиля героя.
- Skeleton.
- Empty state.
- Alert.
- Secondary button.
- Status/role/tier badge semantics.

## Какие локальные решения пришлось принять

1. **Role → Subrole hierarchy.** Design System описывает вложенность навигации как
   открытый вопрос, но не описывает вложенность групп контента внутри Workspace.
   Для Heroes использованы существующие типографические уровни: роль — Sans H3,
   подкласс — Mono Section Title.

2. **Detail popup для информационного профиля.** Design System допускает Detail
   для карточек сущностей, но §2.9 описывает модалку в основном как блокирующее
   решение. Для Heroes используется overlay-профиль, потому что существующих
   компонентов достаточно для первого экрана, но вопрос границы `Modal vs Detail
   Panel/Sheet` остаётся локально отмеченным.

3. **Top counters как один слот метрик.** Compact card ограничена 1-2 ключевыми
   метриками. В Heroes до трёх маленьких counter chips трактуются как один
   смысловой слот `топ-контрпики`, а не как три независимые метрики.

## Какие проблемы действительно обнаружены

- Нужна формализация content hierarchy для случаев `группа → подгруппа → сетка`.
- Нужна ясность по информационному detail-контейнеру: это модалка, панель или
  допустимый вариант detail popup.
- Нужно уточнить, считает ли лимит compact card количество визуальных значений
  или количество смысловых категорий.

## Есть ли реальные основания менять Design System

Сейчас — **нет, изменений Design System не требуется**. Один экран обнаружил
несколько локальных вопросов, но правила задачи требуют не менять Frozen v1.0
заранее. Основание для изменения появится только если те же проблемы повторятся
на нескольких экранах, например на Maps/Players/Roster: тогда можно будет
доказать, что это системная проблема, а не локальная особенность Heroes.
