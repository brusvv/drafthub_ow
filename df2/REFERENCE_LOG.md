## REF-1 — Draft Hub UI Kit (два прохода одного кита), 19.07

**Референс:** внутренний UI Kit "Draft Hub — UI Kit" (Team Analyst), две
итерации — v1 (английский) и v2 (русская локализация + новые компоненты:
Stat Card, Search, Alerts).

### 1. Что делает хорошо
- Явная 3-уровневая лестница поверхностей вместо "фон + один серый".
- 8px spacing-сетка с промежуточными шагами на реальных стыках (20/40 в
  v2) — не произвольный набор чисел.
- Чёткое разделение Neutrals/Semantic в структуре палитры (v2) — сигнал,
  что цвета продуманы как переиспользуемые токены, не подобраны на глаз.
- Toggle-группа и съёмный фильтр-чип показаны в одном блоке, но визуально
  различимы — не смешаны в один паттерн.
- Focused-состояние полей — акцентная обводка, не просто "чуть ярче".
- Stat Card (v2) и Alerts (v2) — компоненты для реальных пробелов, а не
  косметика.

### 2. Что делает плохо / не переносится без потерь
- Таблица показана как основной способ представления списка сущностей —
  для Draft Hub это чужеродный паттерн (см. п.4).
- Степпер (кружки-галочки) несёт только номер шага — недостаточно
  информации для реального сценария живого драфта (см. п.4).
- Типографика без mono-семейства для меты — теряет часть узнаваемости,
  которая уже работает у Draft Hub.
- Hero/Map Card кита рассчитана на одну карточку крупным планом
  (профиль), не на плотную сетку из десятков — прямой перенос сломал бы
  плотность (см. `DESIGN_PHILOSOPHY.md` §3.1).

### 3. Что перенесено в Draft Hub V2
- Лестница поверхностей `--surface-1..4` (§1.1 системы) — формализована
  на основе уже работавшей в V1 палитры, не изобретена заново.
- Spacing-шкала с 20/40 (§1.5).
- Radius-шкала, смещённая к мелким значениям (§1.6).
- Разделение toggle-группы и съёмного чипа как двух явных паттернов
  (§2.3) — было неявным в V1, теперь формализовано.
- Search-инпут (§2.2) — реальный пробел в V1, добавлен.
- Alert как персистентный компонент, отдельный от Toast (§2.9) — реальный
  пробел в V1 (там только toast + разрозненный inline `.error-state`).
- Идея Stat Card — принята с ограничением (см. п.5): только под конкретную
  метрику с действием, не как общий dashboard-виджет (§2.4, "Non-goal"
  из философии §5).

### 4. Что противоречит философии продукта
- **Таблица как основной вид сущностей** — противоречит
  `DESIGN_PHILOSOPHY.md` §3.1 (плотность) и §3.7 (единообразие): завела
  бы второй визуальный диалект той же сущности рядом с уже работающими
  карточками/чипами. Таблица остаётся, но только для admin/служебных
  данных (§2.5 системы).
- **Круговой степпер** — не проходит тест §3.3 (скорость под давлением):
  не показывает команду/действие/карту одновременно, а именно это нужно
  считать за долю секунды во время реального драфата. Остаётся допустимым
  только для линейных мастеров без этой тройной семантики.
- **Отсутствие mono-меты** — противоречит §4 философии (голос и тон,
  узнаваемость) и non-goal "не generic SaaS" (§5) — визуальный характер
  продукта важнее буквального следования кита.

### 5. Изменения, внесённые в DESIGN_SYSTEM.md
- §1.1 (Neutrals), §1.5 (Spacing), §1.6 (Radius) — приняты из кита с
  адаптацией под уже проверенные значения V1, не скопированы буквально.
- §2.2 (Search), §2.3 (два паттерна чипов), §2.9 (Alert отдельно от
  Toast) — добавлены как новые компоненты.
- §2.4 (три уровня плотности карточки) — сформулировано как прямое
  следствие несовместимости кита с плотной сеткой Draft Hub.
- §2.5 (таблица только для admin) и §2.8 (доменный степпер, не generic)
  — зафиксированы как явные отклонения от референса, с обоснованием.

### 6. Статус
Принят с изменениями — токены и часть паттернов взяты, структура экранов
(таблица как основной вид, степпер) отклонена.

### 7. Что затронуто
- `DESIGN_SYSTEM.md` — §1.1, §1.5, §1.6, §2.2, §2.3, §2.4, §2.5, §2.8, §2.9.
- Экраны — пока ни один не спроектирован по итогам разбора; изменения
  на уровне токенов/компонентов, ждут первого реального экрана.

### 8. Lessons learned
- Готовые UI-киты — хороший источник токенов/компонентов (плотность
  проработки выше, чем если придумывать с нуля), но плохой источник
  структуры экрана — структура кита обычно рассчитана на другой продукт
  и переносится с потерями (см. п.4).
- Плотная сетка (Draft Hub) и "одна карточка крупным планом" (типичный
  профильный UI кита) — почти всегда конфликтующие допущения при
  разборе карточных компонентов конкретно, стоит проверять этот пункт
  в любом будущем референсе с готовыми card-компонентами.
