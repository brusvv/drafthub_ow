# Draft Hub V2 — Design System

> Каждое решение здесь выводимо из `DESIGN_PHILOSOPHY.md`. Если меняешь
> токен или компонент — сверься, какому принципу он служит; если ни
> одному — не добавляй его сюда, сначала обсуди.

---

## 0. Анатомия приложения

> Этот раздел объясняет структуру экрана Draft Hub. Здесь решается
> только "что фундаментально", не "как это выглядит в пикселях" —
> пиксельная реализация того, что здесь признано слоем, уже есть в
> §2.11 (навигационная оболочка) и §2.9 (overlay-компоненты).

### 0.1 Метод

Тест на каждый кандидат в слой: **мысленно убрать его — что перестаёт
работать?** Если продукт всё ещё выполняет свою задачу (пользователь
может увидеть текущий раздел и принять решение) — это не фундаментальный
слой, а часть другого, пусть даже важная.

### 0.2 Слои — выведены из реального поведения

#### Shell (Identity + Navigation)
**Ответственность:** говорит пользователю, где он находится (какая
команда, какой раздел) и даёт способ попасть в любой другой раздел.

**Тест на удаление:** без него пользователь не может ни понять, чьи
данные видит, ни переключиться в другой раздел — это единственный путь
смены раздела. Продукт не выполняет свою задачу.
**Фундаментальный слой.**

Во всех рассмотренных сценариях навигация между разделами и
идентификация команды/пользователя видны на каждом экране без
исключения, не меняются между разделами.

Состоит из двух визуально независимых зон (Identity-полоса + Nav-рельс)
— это уже решено и специфицировано в §2.11, здесь не переоткрывается.
Обе зоны имеют общую судьбу (появляются/сокращаются вместе, см. состояния
"живой драфт"/"публичный доступ" в §2.11) — поэтому это один слой с двумя
зонами, а не два раздельных слоя: у них нет сценария, где одна
присутствует без другой.

**Присутствие Shell — не переключатель есть/нет, а спектр минимум из
четырёх точек:**

| Точка спектра | Пример | Механизм |
|---|---|---|
| Полный | Обычная работа | — |
| Урезанный, но в SPA-цикле | Tier List, аноним/Global — mode-switcher сведён к одной некликабельной пилюле, рельс — 1 пункт | Тот же компонент (§2.11), в состоянии с меньшим набором прав |
| Отсутствует, но в SPA-цикле | Auth (вход/регистрация/пользователь без единой команды) | Отдельный экран внутри основного SPA-цикла — тот же жизненный цикл приложения, другой контейнер верхнего уровня |
| Отсутствует, вне SPA-цикла | Tier List, Share Token (`/tier/TOKEN`) | Отдельный маршрут вне основного приложения — рендерится до входа в обычный цикл, Shell-структуры физически нет |

**Критерий присутствия — не факт авторизации, а активный командный
контекст.** Анонимный визит без сессии → Shell урезан, не исчезает
(Tier List/Global); авторизованный пользователь с сессией, но без
единой команды → Shell исчезает полностью. Критерий — не сама
авторизация, а наличие активного командного контекста, которым
Identity-полоса могла бы что-то показать.

Публичный доступ (шеринг-ссылка тир-листа, анонимный визит внутри
приложения, не прямой переход по `/tier/TOKEN`) — не второй отдельный
shell, а урезанное состояние того же слоя (§2.11 уже это реализует:
identity сокращена до логотипа, рельс — один пункт без группировки).
У такого визита нет своей команды/scope/пользователя, которые
Identity-полоса могла бы показать — она не занимает пустые слоты, а не
превращается в отдельную структуру.

#### Workspace
**Ответственность:** содержимое текущего раздела — вся его суть, включая
собственный внутренний процесс/состояние, если у раздела он есть.

**Тест на удаление:** без него есть оболочка, но нет содержимого — у
продукта не остаётся смысла. **Фундаментальный слой.**

**Свидетельство:** Workspace заменяет содержимое активного раздела
целиком при переключении. Внутренние процессы разделов — тоже часть
Workspace, не отдельный слой: пошаговый процесс (например, драфт —
подготовка → баны → результат) переключает содержимое раздела целиком
внутри себя; в разделе администрирования загрузка данных → превью →
импорт → отчёт — та же природа процесса, другой раздел. Оба — это то,
как Workspace конкретного раздела устроен внутри себя, а не
самостоятельные слои приложения.

#### Overlay
**Ответственность:** рендерится поверх Shell и Workspace одновременно,
независимо от того, какой раздел активен, закрывается без ухода со
страницы.

**Тест на удаление:** без него нет способа редактировать/создавать
сущность и нет способа выбрать одну из полусотни вариантов через
picker — но сам цикл "посмотреть раздел → принять решение" по-прежнему
работает. **Фундаментален для конкретных действий (редактирование,
крупный выбор), не для базового цикла просмотра.** Тем не менее
достаточно инфраструктурно значим (принципиально другой механизм
рендера — поверх всего, не внутри Workspace), чтобы быть собственным
слоем, а не паттерном внутри Workspace.

Во всех рассмотренных сценариях модалки редактирования сущностей,
выбор одного варианта из большого набора и информационные попапы
используют один и тот же механизм отображения — поверх Shell и
Workspace одновременно, независимо от того, из какого раздела вызваны.
Различие блокирующий/неблокирующий (модалка vs toast) — свойство
конкретного компонента, уже разобрано на уровне компонента в §2.9, не
повод заводить внутри Overlay ещё под-слои.

### 0.3 Что не является отдельным слоем — и почему

#### "Context Panel" / Inspector / Detail-панель
Не слой — паттерн **внутри** Workspace, реализованный непоследовательно
в двух разных разделах (важная находка сама по себе, не только ответ на
вопрос "слой или нет"):

- **Игроки** и **Состав** — разворачивают подробности **инлайн, внутри
  своего раздела**.
- **Герои** — для концептуально той же задачи ("покажи мне больше про
  эту сущность") используют **overlay-попап** — механически тот же
  слой, что и модалки редактирования, не Workspace-паттерн вовсе.

**Тест на удаление:** убрать инлайн-деталь конкретно у раздела Игроки —
раздел всё ещё выполняет свою задачу (список + редактирование через
модалку) — не фундаментально само по себе.

**Вывод:** это паттерн Workspace, не слой — но два раздела решают одну и
ту же задачу по-разному (инлайн vs overlay), что прямо нарушает
`DESIGN_PHILOSOPHY.md` §3.7 (единообразие вида одной и той же сущности
задачи). Не решаю здесь, какой из двух вариантов правильный — это
решение уровня конкретного экрана, не анатомии; зафиксировано как
открытый вопрос в §5.

#### "Global Actions"
Ни в одном из рассмотренных сценариев не нашлось действия, которое было
бы доступно из Shell независимо от активного раздела — каждое действие
(создать героя, добавить игрока, начать драфт) инициируется изнутри
конкретного Workspace. По `DESIGN_PHILOSOPHY.md`
§3.9 (design debt требует доказательства, не желания) — не завожу слой
под то, чему нет свидетельства в реальном поведении. Если в будущем
появится настоящий сквозной сценарий (например, command palette) — он
получает тот же evidence-based разбор, что и любой другой кандидат в
слой, не вводится заранее "про запас".

### 0.4 Итог

Три слоя, не шесть изначально предполагавшихся — не потому что было
решено урезать список, а потому что рассмотренные сценарии подтверждают
ровно три с чёткой, невзаимозаменяемой ответственностью каждого:
**Shell** (кто/где) → **Workspace** (что, включая внутренний процесс
раздела) → **Overlay** (действие поверх всего, независимо от раздела).
Всё остальное из исходного списка — либо паттерн внутри Workspace
(Context Panel), либо не подтверждено (Global Actions).

### 1.1 Цвет — Neutrals (поверхности и текст)

Лестница элевации в 4 шага — фон отдельно от поверхностей, поверхности
светлеют последовательно. Наследуется из уже проверенной палитры
продукта, формализуется в единую систему:

| Токен | Значение | Назначение |
|---|---|---|
| `--bg` | `#0B0B0D` | Фон приложения |
| `--surface-1` | `#141417` | Базовая поверхность (панель, карточка) |
| `--surface-2` | `#1B1B1F` | Поверхность на поверхности (вложенный блок, попап-хедер) |
| `--surface-3` | `#232326` | Hover/выделенное состояние поверхности |
| `--surface-4` | `#2A2A2E` | Максимальная элевация (модалка поверх модалки — избегать, но токен есть) |
| `--border-1` | `#2F2F36` | Базовая граница |
| `--border-2` | `#3A3A42` | Акцентная/hover граница |
| `--text-1` | `#F5F5F7` | Основной текст |
| `--text-2` | `#A1A1AA` | Вторичный текст |
| `--text-3` | `#71717A` | Третичный текст (мета, подписи, лейблы) |

**Правило:** новая поверхность — всегда следующая ступень лестницы
относительно родителя, не произвольный выбор "что красивее". Карточка на
`--surface-1` содержит вложенный блок на `--surface-2`, не наоборот и не
через ступень.

### 1.2 Цвет — Semantic

| Токен | Значение | Назначение |
|---|---|---|
| `--success` | `#22C55E` | Успешная операция, подтверждение |
| `--warning` | `#FBBF24` | Предупреждение, требует внимания но не блокирует |
| `--danger` | `#EF4444` | Ошибка, деструктивное действие, бан |
| `--info` | `#3B82F6` | Нейтральная информация |
| `--accent` | `#FF8A00` | Primary-действие, tier S, активное состояние |
| `--accent-hover` | `#FFA733` | Hover-состояние accent |

Semantic-цвета **не переизобретаются под конкретный экран** — если нужен
пятый смысл, сначала проверить, не покрывается ли он уже одним из пяти
выше в другой формулировке.

### 1.3 Цвет — Роли и тиры (доменные токены)

Эти два набора — не decoration, это язык самого Overwatch, пользователь
уже знает их значение до входа в продукт. Сохраняются из уже проверенной
семантики продукта без изменений (см. `DESIGN_PHILOSOPHY.md` §3.4 — не
переизобретать работающую семантику):

| Роль | Токен | Значение |
|---|---|---|
| Tank | `--role-tank` | `#4A9EE0` |
| Damage | `--role-damage` | `#E05555` |
| Support | `--role-support` | `#2BBD8E` |

| Тир | Токен | Значение |
|---|---|---|
| S | `--tier-s` | `#F0A030` |
| A | `--tier-a` | `#8BC34A` |
| B | `--tier-b` | `#5B9BD5` |
| C | `--tier-c` | `#888888` |
| D | `--tier-d` | `#A04030` |

### 1.4 Типографика

Два семейства сохраняются сознательно — это часть узнаваемости
продукта, а не случайный выбор (см. `DESIGN_PHILOSOPHY.md` §4 про
non-goal "не generic SaaS"):

- **Sans (Inter)** — контент, который читают: имена героев/карт/игроков,
  заголовки, описания, тела модалок.
- **Mono (Space Mono)** — метаданные, которые сканируют: лейблы полей,
  роли/типы в uppercase+letter-spacing, тайминги, счётчики, технические
  значения (хэши, ID — только в admin-контексте).

Разделение по семейству — это и есть основной сигнал "это для чтения" vs
"это для быстрого сканирования", отдельно от размера/веса.

| Уровень | Семейство | Размер / вес | Трекинг | Назначение |
|---|---|---|---|---|
| H1 | Sans | 32px / 700 | -1% | Заголовок страницы |
| H2 | Sans | 24px / 700 | -0.5% | Заголовок секции |
| H3 | Sans | 20px / 600 | 0% | Заголовок блока внутри секции |
| Section Title | Mono | 12px / 700, uppercase | +8% | Мета-заголовок (лейбл группы карточек, панели) |
| Body Large | Sans | 16px / 400 | 0% | Акцентный текст (имя в карточке) |
| Body | Sans | 14px / 400 | 0% | Базовый текст |
| Body Small | Sans | 13px / 400 | 0% | Вторичный текст в плотных списках |
| Caption | Mono | 11px / 500, uppercase | +4% | Лейблы полей, мелкие метки состояния |

### 1.5 Spacing

8px-сетка, с промежуточными шагами на стыках, где реальная вёрстка чаще
всего в них упирается (принято из разбора референса — см.
`REFERENCE_LOG.md` при первом разборе):

`4 · 8 · 12 · 16 · 20 · 24 · 32 · 40 · 48 · 64`

**Правило:** отступ между однотипными элементами в сетке/списке — всегда
из этой шкалы, не произвольное значение "на глаз". Если ни одно значение
шкалы не подходит — это сигнал, что элементы на самом деле разнотипные и
нужен другой паттерн, не повод добавить одиннадцатое число в шкалу.

### 1.6 Радиусы

`4px · 6px · 8px · 12px · 24px · Full`

Мелкие радиусы (4-8px) — для интерактивных элементов плотного слоя
(кнопки, чипы, инпуты, строки таблиц). `12px` — для карточек/панелей.
`24px`/`Full` — для пилюль/бейджей и круглых элементов (аватар-плейсхолдер,
close-кнопка).

### 1.7 Тени и элевация

Тени — не декоративные, только для реально всплывающих поверх контента
элементов (модалка, тултип, dropdown), не для обычных карточек в сетке
(там элевация — через `--surface-N`, не через тень):

| Токен | Значение | Назначение |
|---|---|---|
| `--shadow-popover` | `0 8px 24px rgba(0,0,0,.4)` | Dropdown, тултип |
| `--shadow-modal` | `0 20px 50px rgba(0,0,0,.55)` | Модалка |

### 1.8 Motion

Минимальный набор — движение обслуживает считываемость состояния, не
развлекает. Анимация оправдана только для одного из трёх: подтвердить
действие пользователя, объяснить изменение состояния (появился/исчез
элемент, сменилась вкладка), помочь взгляду не потерять место при
переходе. Если анимацию нельзя отнести ни к одному из трёх — она лишняя,
даже если выглядит красиво на статичном скриншоте.

| Токен | Значение | Назначение |
|---|---|---|
| `--duration-fast` | `100ms` | Hover, нажатие, переключение чекбокса |
| `--duration-base` | `150ms` | Появление/исчезновение блока, смена вкладки |
| `--duration-slow` | `250ms` | Модалка/попап открытие-закрытие |
| `--ease-out` | `cubic-bezier(.16,1,.3,1)` | Стандартная кривая появления |

**Правило:** ничего не анимируется дольше `--duration-slow`. Если
анимация "красивая, но долгая" — это конфликт с `DESIGN_PHILOSOPHY.md`
§3.3 (скорость под давлением), не эстетическое решение.

## 2. Компоненты

Для каждого — назначение, состояния, где именно применять именно этот
компонент (не соседний похожий).

### 2.1 Кнопки

Четыре семейства, три состояния (Default/Hover/Disabled) + отдельно
icon-button (квадрат, только иконка, для компактных панелей инструментов):

- **Primary** — одно на экран/секцию, обязательное следующее действие
  (см. `DESIGN_PHILOSOPHY.md` §3.3).
- **Secondary** — альтернативное, необязательное действие рядом с Primary.
- **Ghost** — третичное действие, не соревнуется визуально с первыми
  двумя (закрыть, отменить, "показать ещё").
- **Danger** — необратимое/деструктивное (удалить, забанить, сбросить).

**Do:** одна Primary-кнопка на видимую область экрана.
**Don't:** две Primary рядом — если оба действия равнозначны, обе должны
быть Secondary, конфликт решается позицией/порядком, не дублированием
акцента.

### 2.2 Поля ввода

Text / Select / Search — единый визуальный язык, различаются только
иконкой и поведением. Focused — акцентная обводка (не просто чуть ярче
серая), достаточно контрастная для быстрого визуального поиска "куда я
сейчас печатаю" (важно при быстром вводе данных аналитиком).

Search — с иконкой лупы и мгновенным фильтром списка при вводе
(debounce не длиннее `--duration-base`), не только по Enter — в плотных
списках (герои/карты/игроки) это единственный быстрый способ найти
конкретную сущность среди десятков карточек.

### 2.3 Чипы — два разных паттерна, не путать

Ключевое разделение, явно проверенное при разборе референса и
подтверждённое реальным использованием в Draft Hub (баны/фильтры):

- **Toggle-группа** (All/Tank/Damage/Support) — один активный элемент из
  набора, взаимоисключающие. Визуально: заливка на активном, без крестика.
- **Съёмный фильтр-чип** (Control ×, Lijiang Tower ×) — несколько
  одновременно активных, каждый убирается независимо. Визуально: контур +
  крестик, список активных отдельно виден от источника выбора.

**Правило:** если элементов может быть выбрано больше одного одновременно
— это съёмный чип, не toggle. Смешивать паттерны в одном фильтр-блоке
без разделения — ошибка, пользователь не должен угадывать, можно ли
выбрать несколько.

### 2.4 Карточки — три уровня плотности одной сущности

Прямое следствие `DESIGN_PHILOSOPHY.md` §2 и §2.5 — одна и та же
сущность (герой/карта) существует в разных плотностях, не в разных
визуальных языках:

1. **Compact (сетка)** — портрет/баннер + имя + 1-2 ключевые метрики
   максимум (тир, роль). Это то, что видно, когда сущностей на экране
   десятки — решение "открывать/не открывать", не полный профиль.
2. **Detail (попап/панель)** — полный профиль: все метрики, контрпики/
   синергии, сила на картах, заметки. Открывается по клику из compact,
   не существует отдельно в сетке.
3. **Stat Card (сводка, используется точечно)** — число + тренд + подпись
   периода. **Не** для сущностей (герой/карта/игрок) — только для
   агрегированных метрик самого продукта (см. `DESIGN_PHILOSOPHY.md` §5,
   non-goal "не dashboard ради dashboard" — Stat Card появляется только
   там, где сводка меняет следующее действие, например "3 карты не
   оценены" рядом с кнопкой "Оценить").

**Don't:** добавлять в compact-карточку данные "раз уж есть место" — если
карточка требует внутреннего скролла или уменьшения шрифта, чтобы всё
влезло, — это сигнал вынести добавленное в detail, а не сжимать (см.
§2.5 философии).

### 2.5 Таблицы — только для admin/служебных данных

Основная сетка сущностей (герои/карты/игроки) — **не** таблица, остаётся
карточками/чипами (это существующий и правильный визуальный язык Draft
Hub, см. критику референса). Таблица — узкий компонент, только для:
списка пользователей команды, CSV-превью импорта, журнала действий.
Не вводить табличное представление как альтернативный вид для
героев/карт "для тех, кто предпочитает таблицы" — это второй визуальный
диалект той же сущности, прямое нарушение §3.7.

### 2.6 Бейджи

- **Tier badge** (S/A/B/C/D) — цвет из §1.3, всегда с буквой внутри, не
  только цветом (доступность).
- **Role badge** (Tank/Damage/Support) — иконка + цвет из §1.3.
- **Status badge** (текстовый, например "NEW", "БАН") — semantic-цвет из
  §1.2, не доменный.

### 2.7 Вкладки

Подчёркивание активной вкладки, без заливки фона — плоский, не
соревнуется с чипами/кнопками за визуальный вес. Для навигации верхнего
уровня продукта — см. §3.4 (разделение primary/secondary навигации), не
для этого компонента (вкладки — внутри одного раздела).

### 2.8 Индикатор шагов драфта (доменный компонент, не generic stepper)

Прогресс живого драфта (бан/пик/сторона по каждой карте) — не абстрактный
степпер "шаг 3 из 5". Каждый шаг несёт минимум 3 факта одновременно:
какая команда действует, что за действие (бан/пик/сторона), к какой
карте это относится — и это должно считываться за долю секунды во время
реального драфта под таймером (см. `DESIGN_PHILOSOPHY.md` §3.3). Шаги
одной карты визуально группируются вместе (общая подложка/линия), не
существуют как одинаковые несвязанные точки в ряд — иначе на Bo5/Bo7 (до
17 шагов) невозможно быстро понять, где заканчивается одна карта и
начинается следующая.

**Не использовать** декоративный кружки-с-галочками степпер (из
референса) для этого компонента — он передаёт только номер шага, не
команду/действие/карту одновременно, и потому не проходит тест §3.3.
Круговой степпер остаётся допустимым **только** для линейных мастеров
без этой тройной семантики (например, настройка нового импорта).

### 2.9 Модалка / Тултип / Alert / Toast — разная стойкость сообщения

Четыре разных компонента для разной "стойкости" информации, не
взаимозаменяемы:

- **Toast** — эфемерное подтверждение ("Сохранено"), исчезает само,
  не требует реакции.
- **Alert** — персистентный инлайн-баннер (успех/предупреждение/ошибка
  импорта), остаётся видимым пока не закрыт вручную или контекст не
  сменился — для всего, что пользователь может пропустить если оно
  исчезнет само (ошибка импорта строки, не просто "сохранено").
- **Tooltip** — контекстная подсказка по hover/focus, не для критичной
  информации (может не быть замечена).
- **Модалка** — блокирующее решение (подтверждение удаления, редактирование
  сущности) — используется только когда решение действительно должно
  прервать текущий контекст, не как контейнер "чтобы не загромождать
  страницу".

### 2.10 Пустое / загрузка / ошибка — обязательная тройка на раздел

Каждый раздел, зависящий от внешних данных (Supabase/Sheets/каталог
портретов), обязан реализовать все три состояния явно, не только
happy path:

- **Loading** — skeleton, повторяющий форму реального контента (не
  спиннер по центру пустого экрана).
- **Empty** — конкретная причина + конкретное действие (см.
  `DESIGN_PHILOSOPHY.md` §3.2), не просто "ничего не найдено".
- **Error** — безопасный текст (никогда не сырое сообщение об ошибке
  без экранирования — эта категория ошибок уже приводила к реальной
  уязвимости) + повторить-действие.

### 2.11 Навигационная оболочка (первый спроектированный экран, 19.07)

Архитектура и разбор масштабируемости — §5.1. Здесь — сама реализация.

**Структура — два независимых яруса, не один блок:**

- **Identity-полоса** (верх, во всю ширину) — команда/scope
  (Global/Team/Personal)/пользователь. Высота `56px`, `--surface-1`,
  border-bottom `1px solid var(--border-1)`. Не меняется ни от вкладки,
  ни от состояния драфта — прямое следствие §3.8 (память).
- **Рельс** (слева, во всю высоту под identity-полосой) — контентная
  навигация. Ширина `220px` развёрнутый / `64px` свёрнутый,
  `--surface-1`, border-right `1px solid var(--border-1)`.
  `width var(--duration-base) var(--ease-out)`.

**Пункт рельса:** высота `40px`, padding `0 16px`, gap иконка-лейбл
`10px`, border-radius `8px` (интерактивный элемент, §1.6), лейбл Body
14px (§1.4).

| Состояние | Фон | Иконка | Лейбл | Доп. сигнал |
|---|---|---|---|---|
| По умолчанию (Primary) | прозрачный | outline, `--text-2` | `--text-1`, 400 | — |
| По умолчанию (Secondary) | прозрачный | outline, `--text-3` | `--text-2`, 400 | — |
| Hover | `--surface-2` | `--text-1` | `--text-1` | — |
| Активный (текущий раздел) | `--surface-3` | **filled**-вариант, `--accent` | `--text-1`, 600 | левая полоса `2px`, `--accent` |
| `:focus-visible` | как hover | — | — | `outline: 2px solid var(--accent); outline-offset: 2px` |

Активное состояние — три независимых сигнала одновременно (фон + смена
формы иконки outline→filled + жирность лейбла + полоса), не только цвет
— прямое следствие §3.6: если убрать `--accent` совсем (color-blind
режим/принтер), активный пункт всё равно читается.

**Группировка:** Primary (Карты/Герои/Драфт/Состав/Игроки) сверху,
разделитель (`1px solid var(--border-1)`, margin `16px`), Secondary
(Tier List/Настройки/Админ) снизу — визуально тише через `--text-2`/
`--text-3` по умолчанию (не через уменьшенный размер — размер шрифта
кодирует важность контента, не свою группу навигации, тут используется
одна из двух Neutrals-ступеней текста, уже определённых в §1.1, не новый
токен, — то самое переиспользование существующего смысла из §3.4/§3.9).

**Состояние: живой драфт (вход в раздел Драфт).** Рельс сворачивается
`220px→64px`. Secondary-группа и разделитель **скрываются полностью**,
не сжимаются до иконок — во время реального драфта под таймером даже
неактивная иконка Secondary — периферийный шум, который проверяется
§3.3 буквально ("всё остальное — шум"). Primary остаётся иконками без
лейблов, активный пункт (Драфт) сохраняет все 3 сигнала состояния из
таблицы выше. Identity-полоса не меняется (§3.8).

**Состояние: публичный доступ (открыта шеринг-ссылка тир-листа, нет
сессии).** Identity-полоса — только логотип/название продукта, без
команды/scope/пользователя (нечего показывать, не оставлять пустые
слоты). Рельс — один пункт, Tier List, без группировки Primary/Secondary
(группировка не нужна, когда пункт один) и без разделителя.

### 2.12 Что здесь НЕ решено — см. §5.1

Второй уровень вложенности (если у будущего раздела появятся свои
вкладки) и модель usage moments за пределами трёх текущих персон
(`DESIGN_PHILOSOPHY.md` §2) — сознательно не спроектированы сейчас, это
не пробел реализации, а явное решение не проектировать под
несуществующую пока функциональность (§5.1).

## 3. Паттерны

### 3.1 Плотность по контексту, не по вкусу
Сетка сущностей — compact по умолчанию (§2.4). Единственная страница,
где допустима более просторная карточка — это одностраничный detail-вид,
не список.

### 3.2 Навигация — primary/secondary, не плоский список вкладок
Рабочий контур (Карты/Герои/Драфт/Состав/Игроки) визуально отделён от
вторичного (Tier List/Настройки/Админ) — прямое следствие того, что во
время реальной работы пользователь переключается почти всегда внутри
первой группы, вторая — редкие визиты. Реализация: боковой рельс,
Primary сверху/Secondary снизу через разделитель, тонкая identity-полоса
сверху (команда/scope/пользователь) вне рельса — не смешана с контентной
навигацией. См. §5.1 (архитектурное решение и разбор масштабируемости) и
навигационную оболочку как первый спроектированный экран.

### 3.3 Живой драфт — состояние всегда важнее списка возможностей
Во время шага драфта интерфейс сначала показывает "что происходит
сейчас" (кто действует, что за действие), и только потом — доступные
варианты для выбора. Порядок важен: варианты без контекста действия —
источник ошибок под давлением таймера.

### 3.4 Цвет команды — последовательный на протяжении всего драфта
Если команда A получает цвет в момент выбора формата — тот же цвет
сопровождает её действия (бан/пик/сторона) до конца серии, не
переприсваивается по ходу. Пользователь не должен пересчитывать "какая
команда сейчас какая" на каждом шаге.

## 4. Доступность — baseline, не отдельный чек-лист

- `:focus-visible` — обязателен на каждом интерактивном элементе с
  момента создания компонента, не патч постфактум.
- Состояние никогда не кодируется только цветом — тир-бейдж всегда с
  буквой, статус-иконка всегда с текстом рядом при первом появлении в
  контексте (не только в тултипе). **Уровень применения — контекстный**
  (подтверждено на 3 независимых экранах, `META_VALIDATION_REPORT.md`
  §1.3/§2.2): если состояние разное у соседних элементов (шаг
  сделан/текущий/впереди в Драфте) — сигнал нужен на каждом; если
  состояние одно и то же для целого региона (весь тир-лист read-only) —
  одного сигнала на регион достаточно, повтор на каждом элементе — шум,
  не более безопасный вариант. Первая версия мокапа Tier List добавляла
  замок на каждую read-only пилюлю буквальным прочтением правила — это
  оказалось гиперкоррекцией, не более правильным применением; убрано
  после визуальной проверки, оставлен один баннер на экран.
- Контраст текста и активного/фокусного состояния — минимум WCAG AA
  на всех новых токенах из §1.

### 5.1 Навигация — архитектурное решение и разбор масштабируемости (19.07)

Из четырёх рассмотренных архитектур (верхняя панель в одну строку;
постоянный полновысотный сайдбар; верхняя панель с Secondary в overflow;
тонкая identity-полоса + рельс с раздельным поведением для живого
драфта) выбрана последняя. Обоснование — единственная из четырёх,
которая отвечает на `DESIGN_PHILOSOPHY.md` §3.3 прямым, спроектированным
поведением (сворачивание рельса при входе в Драфт), а не косвенно
("не мешает"). Реализация — §2.11.

**Проверено против роста продукта (гипотетические разделы горизонта
2-3 лет: аналитика матчей, подготовка к сопернику, AI-инструменты,
библиотека стратегий, тренировки):**

- **Механизм (рельс + сворачивание) — устойчив к росту.** Выведен из
  §3.3, которое не устареет от добавления разделов; вертикальный список
  растёт строками, не упирается в горизонтальную ширину как вариант "А"
  из разбора.
- **Таксономия (ровно 2 корзины, 1 уровень) — НЕ гарантированно
  устойчива.** Прогон пяти гипотетических разделов через `DESIGN_PHILOSOPHY.md`
  §2 показал: 3 из 5 не укладываются в модель чисто — не потому что нав
  плоха, а потому что это либо новые usage moments, которых три текущие
  персоны §2 не описывают (аналитика матчей — сценарий "после", не "до/
  во время/между"; тренировки — вообще другая персона, не про подготовку
  команды), либо функциональность, которая не должна быть отдельным
  разделом навигации вообще (AI — capability внутри существующих
  экранов, не destination; см. §3.9, design debt требует доказательства).
  Библиотека стратегий — единственный из пяти чистый кандидат как есть.
- **Явные компромиссы, принятые сейчас:** второй уровень вложенности
  навигации не спроектирован (см. открытый вопрос выше) — по методологии
  §6 философии не проектируется под функциональность, которой ещё нет.
  Это не забывчивость, а стоимость следования собственному процессу.
- **Пересматривать когда:** появится реальный экран для раздела, не
  укладывающегося в текущую модель персон — тогда сначала обновляется
  `DESIGN_PHILOSOPHY.md` §2 (появление нового usage moment), потом, если
  понадобится, таксономия рельса §2.11 — не наоборот.
