# Implementation Validation — экран «Карты» (mockup)

> Проверка реализации `maps-screen-mockup.html` против утверждённого
> `maps-screen-spec.md`. Спецификация, `DESIGN_PHILOSOPHY.md` и
> `DESIGN_SYSTEM.md` в процессе реализации не изменялись.

---

## Что реализовано

- Shell с активным разделом «Карты».
- Workspace header с названием, meta текущего scope и спокойным статусом задач.
- Action summary из трёх actionable Stat Cards: `Без оценки`, `Слабые карты`,
  `Мало данных`.
- Toolbar: Search, single-select toggle типов карт, Select сортировки.
- Compact map grid с группировкой по типу карты.
- Состояния: default, loading, empty catalog, empty filter, error, detail.
- Detail popup / Overlay с сильными героями, рисками, заметками по сторонам и
  Secondary/Ghost действиями.

---

## Какие правила Frozen v1.0 выдержаны

- Новые токены, компоненты и варианты компонентов не добавлялись.
- Таблица не использовалась для карт как доменных сущностей; выбран compact grid.
- Action summary содержит только метрики, ведущие к действию, а не декоративные KPI.
- Loading повторяет форму контента через skeleton.
- Empty catalog и empty filter разведены, потому что требуют разных действий.
- Error реализован как persistent Alert с безопасным текстом и retry action.
- Focus-visible задан на интерактивных элементах mockup.
- Цвет не является единственным носителем состояния: status badges содержат текст.

---

## Локальные решения реализации

1. **Demo toolbar.** Панель переключения состояний нужна только для проверки
   mockup и не является частью экрана Maps.
2. **Текстовые nav-иконки.** В mockup использованы простые glyph-иконки, чтобы не
   вводить новый icon set и не проектировать графику вне Design System.
3. **Placeholder-баннеры карт.** Реальные screenshots карт не добавлялись: mockup
   проверяет композицию, плотность и состояния, а не asset pipeline.
4. **Retry simulation.** Кнопка `Повторить загрузку` в demo возвращает в loading,
   затем default; это имитация поведения, не продуктовая логика.

---

## Проблемы, обнаруженные реализацией

- **Icon language для nav и map type пока не формализован.** Для mockup это не
  блокер, потому что статус/текст дублируют смысл, но реальная реализация должна
  использовать утверждённый набор иконок или отдельно валидировать его.
- **Map screenshots/asset ratio требуют продуктового решения.** В mockup выбран
  стабильный баннерный placeholder; реальная реализация должна проверить, есть ли
  у источника карт единый aspect ratio.
- **Action summary needs data thresholds.** Визуально компонент достаточен, но
  продукт должен определить пороги `Слабая карта` и `Мало данных`.

---

## Нужно ли менять Design System или спецификацию

Нет. Реализация не выявила необходимости менять Frozen Design System или
`maps-screen-spec.md`. Найденные вопросы относятся к данным/ассетам/иконкам и
должны решаться отдельными задачами, а не правкой системы по ходу mockup.
