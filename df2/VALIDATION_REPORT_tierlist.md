## 1. Global/Public — один режим данных, реализовано

Подтверждено в реальном коде на прошлом шаге (`renderPublicMode()`
жёстко ставит `_applyTierMode('global')`; `renderAppModeSwitcher()` для
анонима — некликабельный `<div>`, не попап). В макете: «Аноним» — не
пятая кнопка тулбара, а чекбокс, который **включается только когда
активен Global** (задизейблен на Командном/Личном/Share-ссылке).

При включении: mode-switcher сворачивается в статичную плашку `🌐`,
`identity-right` заменяется на кнопку «Войти» вместо
scope-switch/team-chip, rail схлопывается до одного пункта (Tier List —
группировать больше нечего, тот же паттерн, что уже был в
`nav-shell-mockup.html`, просто впервые применён на реальном экране, а
не в изолированной демонстрации). Shell **не исчезает** — только
урезается, что и было целью находки.

**В процессе реализации нашёл и исправил баг в первой версии разметки:**
класс-маркер `.tierlist-rail-item` изначально стоял внутри
`.rail-secondary`, а CSS-правило схлопывания прятало `.rail-secondary`
целиком без исключения — в первом скриншоте rail становился полностью
пустым вместо "один пункт остаётся". Поймано визуальной проверкой через
headless Chrome, не было бы видно при чтении одного кода.

## 2. Share Token — отдельный экран без Shell, реализовано

`handleSharedTierUrl()` в реальном V1 подменяет `document.body.innerHTML`
целиком — не рендерится ВНУТРИ приложения вообще. В макете это не
состояние `.frame`, а отдельный блок `#sharedStandalone`: без
`identity-bar`, без `rail`, только сам тир-контент по центру страницы.
Визуально это должно бросаться в глаза при сравнении со всеми
остальными состояниями — по скриншотам это подтвердилось: единственный
экран разбора, где нет узнаваемой рамки продукта вообще.

**Открытый архитектурный вопрос остаётся, не закрыт этим макетом:**
`DESIGN_SYSTEM.md` §0.3/§0.4 описывает три слоя (Shell/Workspace/
Overlay) как исчерпывающую модель компоновки экрана. Share Token в эту
модель не попадает ни одним из трёх — это не Overlay (не поверх
Workspace, а вместо всего), не Workspace (нет Shell вокруг, которому
Workspace обычно принадлежит). Реализовать в макете можно было и без
ответа на этот вопрос (что и сделано), но сам вопрос предложением из
трёх слоёв не закрывается — фиксирую отдельно, не выдаю мокап за ответ
на него.

## 3. Замки на пилюлях — убраны, реализовано

Было: `.tier-row[data-editable="false"] .tier-pill::after{content:'🔒'}`
на каждой пилюли read-only-ряда. Убрано полностью — остались только
баннер (текстовый дубль состояния целиком для экрана), `cursor:default`
и `opacity:.88`. Проверено на состоянии «Глобальный» (5 тиров × до 3
элементов) — баннер сверху читается сразу, повторение на каждой пилюле
не добавляло ничего нового, только визуальный шум на плотном списке.

## 4. Новое, найдено по ходу реализации: рассинхрон имён токенов ролей

Не было частью трёх находок с прошлого шага — обнаружено при переносе
CSS в исправленный файл. `tierlist-screen-mockup.html` использовал
`--tank`/`--damage`/`--support` (легаси-имена V1), тогда как
`draft-screen-mockup.html` и `DESIGN_SYSTEM.md` §1.3 используют
`--role-tank`/`--role-damage`/`--role-support`. Значения совпадали
(коллизии не было), только имя переменной — но это именно то, от чего
предостерегает `DESIGN_PHILOSOPHY.md` §3.7 (единообразие): один и тот
же токен, разные имена в разных файлах — риск разойтись по значению
при следующей правке одного без другого.

**Исправлено по месту** (переименовано на канон), не заведено отдельным
вопросом — расхождение было чисто техническим, не архитектурным
решением, которое стоило бы обсуждать.

## Что НЕ стало находкой (проверено и подтвердило систему)

- Mode-switcher как компонент, принадлежащий Workspace, не Shell (уже
  установлено раньше) — подтвердилось повторно: он корректно живёт
  внутри `.workspace`, Shell (identity-bar/rail) не знает о существовании
  тир-режимов вообще, как и должно быть.
- Формат share-ссылки (`draftub.app/tier/TOKEN`) — использован
  одинаково и в панели «Поделиться» (Personal), и в отдельном демо-
  состоянии Share Token — визуально подтверждает, что это один и тот же
  механизм с двух сторон (создание ссылки / переход по ссылке), не два
  разных.
