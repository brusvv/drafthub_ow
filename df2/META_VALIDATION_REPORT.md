> посмотреть на все четыре разбора (`VALIDATION_REPORT_heroes.md`,
> `_draft.md`, `_tierlist.md`, `_auth.md`) как на одно целое. Порядок

> оказалось ложным, что остаётся открытым. Предложений по правкам

## 1. Подтвердилось несколько раз — не единичное наблюдение

### 1.1 Shell — не бинарный (есть/нет), а спектр из минимум трёх разных точек

Это главный результат синтеза — по отдельности каждый отчёт нашёл
только одну точку этого спектра и не называл это спектром:

| Экран | Что с Shell | Механизм |
|---|---|---|
| `nav-shell-mockup.html` (§2.11, до screen-разборов) | Полный → свёрнут до иконок → полностью пуст (rail 1 пункт) | Один компонент, три CSS-состояния |
| Tier List, аноним + Global | Урезан (mode-switcher → одна некликабельная пилюля, rail → 1 пункт) | Тот же компонент, тот же SPA-цикл, просто меньше прав |
| Auth (login/register/no-teams) | Полностью отсутствует в разметке (не свёрнут — не отрендерен) | **Другой** контейнер, sibling `#authScreen`/`#app`, toggle `.style.display`, тот же SPA-цикл |
| Tier List, Share Token (`/tier/TOKEN`) | Полностью отсутствует | `document.body.innerHTML` целиком, **вне** SPA-цикла, до `initSession()` |

Каждый отчёт по отдельности сформулировал это как «Shell тут
отсутствует» — верно локально, но синтез показывает, что «отсутствует»
на самом деле три архитектурно разных случая, не один паттерн,
применённый трижды. `VALIDATION_REPORT_auth.md` (п.3) это уже заметил
для пары Auth/Share Token — синтез добавляет третью точку (урezанный,
не отсутствующий Shell у Tier List/аноним) в ту же ось.

### 1.2 Критерий присутствия Shell — не сессия, а активный командный контекст

Подтверждено дважды, независимо, на разных осях:
- Tier List: анонимный визит (нет сессии вообще) → Shell урезан, не
  исчезает.
- Auth, `no-teams`: сессия ЕСТЬ (доступна кнопка «Выйти»), команд — 0 →
  Shell исчезает полностью.

Если бы критерий был «есть сессия», два этих случая противоречили бы
друг другу (у первого сессии нет, Shell частично есть; у второго
сессия есть, Shell нет вообби). Единственная формулировка, под которую
подходят оба независимо найденных случая: Shell зависит от наличия
**активного командного контекста**, не от факта авторизации самой по
себе. Это не гипотеза одного отчёта, поддержанная другим — это
формулировка, которая _выводится только из пересечения двух_,
по отдельности каждый допускал более простое (и, как выяснилось,
неверное) прочтение.

### 1.3 «Не только цвет» (`DESIGN_PHILOSOPHY.md` §3.6) — принцип держится, но уровень применения оказался контекстным

Три независимых применения:
- Героев: priority-индикатор — число рядом с точками.
- Драфт: шаг-чип — рамка + непрозрачность + позиция, три сигнала НА
  КАЖДОМ элементе (у соседних элементов разное состояние).
- Tier List: баннер read-only — один текстовый сигнал НА ЭКРАН, замки
  на отдельных пилюлях признаны избыточными и убраны.

Принцип не опровергнут ни разу — но третий случай показывает, что «не
только цвет» не означает «сигнал на каждом элементе»: когда состояние
ОДНО и то же для целого региона (весь тир-лист read-only), достаточно
одного сигнала на регион; когда состояние РАЗНОЕ у соседних элементов
(шаг сделан/текущий/впереди), сигнал нужен на каждом. Это уточнение
области действия правила, не новое правило и не отмена старого.

### 1.4 Токены — дисциплина реально нужна, не бюрократия ради бюрократии

Два независимых случая drift, оба поймались до того как стали
проблемой пользователя:
- Драфт: `--team-a`/`--team-b` заранее сверены со всем существующим
  списком цветов перед добавлением — предотвращение.
- Tier List: `--tank`/`--damage`/`--support` (легаси-имена) против
  канона `--role-tank`/`--role-damage`/`--role-support` — разошлись
  ИМЕНА при совпадающих значениях, поймано при переносе кода, не по
  жалобе пользователя.

Один — предотвращение до факта, другой — обнаружение после факта, но
оба про одно: `DESIGN_PHILOSOPHY.md` §3.7 (единообразие) — не
формальность, drift возникает быстро и без злого умысла даже за 1-2
дня работы над 4 экранами.

### 1.5 Существующие паттерны, подтверждённые независимо больше одного раза (без изменений)

- Toggle-группа чипов (§2.3) — Герои (фильтр по роли) и Tier List
  (фильтр по типу карты) — оба раза подошла без правок.
- Primary-кнопка «одна на экран/секцию» (§2.1) — Auth (Войти/Создать
  аккаунт/Создать команду — три экрана, три раза правило проверено) и
  неявно везде остальные (никто не нарушил).
- `DESIGN_PHILOSOPHY.md` §3.3 (скорость под давлением) как реальный,
  work-load-bearing тест, не декларация — явно провалил через него один
  паттерн-кандидат (круговой степпер, ещё на этапе `REFERENCE_LOG.md`,
  до экранов) и один раз подтвердил корректность решения (Secondary-
  кнопки, не плитки, для бинарного выбора стороны в Драфте).

## 2. Оказалось ложным — не просто «уточнено», а обратное первоначальному предположению

### 2.1 «Отсутствие Shell — один паттерн» — ложно

Неявное, никогда явно не сформулированное, но действовавшее по
умолчанию предположение (до Auth-разбора): раз Share Token уже показал
«экран без Shell», второй такой случай (Auth) скорее всего будет
устроен так же технически. `VALIDATION_REPORT_auth.md` (таблица в п.3)
показывает три независимых расхождения (механизм переключения,
переиспользование токенов/классов, место в жизненном цикле SPA) — не
вариации одной темы, а два не связанных друг с другом технических
решения с общей ПРИЧИНОЙ, но разной РЕАЛИЗАЦИЕЙ. Одинаковая причина
не означает одинаковый механизм — этого само по себе не гарантирует
никакая явная формулировка в текущих документах.

### 2.2 «Замок на каждом read-only элементе — правильное прочтение §3.6» — ложно

Первая версия `tierlist-screen-mockup.html` (до вмешательства
пользователя) добавляла замок на каждую пилюли read-only-ряда как
буквальное прочтение «состояние не только цветом». Это оказалось
избыточным — не нарушением правила, а его гиперкоррекцией: правило
удовлетворяется один раз на экран, повтор на 20+ элементах не добавляет
информации, только шум (конфликт с `DESIGN_PHILOSOPHY.md` §3.1).
Ложной была не сама идея редундантного сигнала, а презумпция, что
редундантность должна быть на уровне КАЖДОГО элемента по умолчанию.

### 2.3 «Global и Public — два раздельных режима тир-листа» — ложно

Первая версия мокапа (до этого же разговора) моделировала «Публичный»
как четвёртую равноправную кнопку тулбара рядом с Командный/Глобальный/
Личный. Проверка реального кода (`renderPublicMode()`,
`renderAppModeSwitcher()`) показала: это один и тот же режим данных
(`_applyTierMode('global')` жёстко), различие только в правах доступа
к UI вокруг него. Ложным было не наблюдение «аноним видит другой UI» —
а вывод из него, что это ОТДЕЛЬНЫЙ режим, а не урезанный вариант
существующего.

## 3. Остаётся открытым — явно, не решаю здесь

### 3.1 Ось «внутри SPA-цикла vs отдельный маршрут» — не покрыта текущей моделью слоёв

`DESIGN_SYSTEM.md` §0.5 фиксирует три слоя (Shell/Workspace/Overlay)
как исчерпывающую модель. Синтез показывает: этой модели не хватает
ортогональной оси — не «какой слой», а «работает ли вообще SPA-цикл
на этом экране». Share Token — не Overlay (не поверх Workspace, а
вместо всего), не Workspace-без-Shell (нет Shell вокруг, которому
Workspace обычно принадлежит бы) — он вне модели целиком, по другой
причине, чем «слой не применили». Найдено дважды с разных сторон
(`_tierlist.md` п.2, `_auth.md` п.4), формулировка ни разу не
предложена — только зафиксирован факт нехватки.

### 3.2 Спектр Shell (см. 1.1) — сам по себе не формализован

Три найденные точки спектра (урезан / отсутствует-но-в-цикле /
отсутствует-вне-цикла) — наблюдение синтеза, не результат ни одного
отдельного отчёта. Не факт, что три точки — все существующие; не факт,
что нужна формальная шкала, а не просто три независимых case study.
Открыто, не предрешено.

### 3.3 Team A/B — цвет не выбран, решение отложено с 3 вариантами на столе

`VALIDATION_REPORT_draft.md` п.2 — синий/красный интуитивно нагляднее,
но подтверждённо коллизирует с action-цветом БАН (`--danger`), не
только с ролевыми цветами, как предполагалось раньше. Ни один из трёх
вариантов (оставить фиолетовый/розовый · сменить action-цвет бана ·
два разных токена для экранов с ролями и без) не выбран. Единственный
вопрос из четырёх отчётов, где V1-код НЕ проверен с уверенностью
(есть основание полагать похожее напряжение уже было в V1
`ourBans`/`enemyBans`, но не подтверждено построчно).

### 3.4 §2.4 (карточка, 1-2 метрики) — считает значения или смысловые категории?

`VALIDATION_REPORT_heroes.md` п.1. Один case (топ-контрпики, 3
визуальных элемента внутри одного смыслового слота) не даёт достаточно
данных чтобы решить в общем виде — нужен ли второй, структурно похожий
пример на другом экране, прежде чем формулировать правило точнее, или
хватит одного прецедента.

### 3.5 Вложенность контента (не навигации) — второй независимый случай той же формы вопроса

`VALIDATION_REPORT_heroes.md` п.4 сам называет это «той же формы, что
открытый вопрос навигации» (`DESIGN_SYSTEM.md` §5.1, до всех
screen-разборов). Теперь у вопроса два независимых проявления на разных
осях (навигация vs контент) — не решает вопрос, но поднимает его вес:
если третий независимый случай найдётся на ещё одной оси, это уже
паттерн, требующий одного общего решения, а не два отдельных.

### 3.6 Detail-попап — Модалка или неописанная категория?

`VALIDATION_REPORT_heroes.md` п.5. §2.9 определяет Модалку через
«блокирующее решение» — detail-попап героя ближе к информационному
контейнеру. Использован `.modal-overlay` прагматично (другого крупного
контейнера в системе нет), вопрос не закрыт этим выбором. Единственный
из открытых вопросов, не встретившийся повторно ни на одном другом
экране — не значит менее важный, значит меньше данных для синтеза.

### 3.7 Search (§2.2) — предписан документом, не проверен на масштабе

`VALIDATION_REPORT_heroes.md` п.3. Добавлен в мокап «как написано в
правиле» — не проверено, была ли сама формулировка правила рассчитана
на текущий масштаб (~40 героев) или на будущий рост. Единственный
открытый вопрос про масштабируемость самого правила, не про его
трактовку.

### 3.8 БАН vs BAN — не открытый вопрос системы, но не закрыт как вопрос к пользователю

`VALIDATION_REPORT_draft.md` п.1 — `DESIGN_PHILOSOPHY.md` §4 однозначно
отвечает (кириллица), решение не менялось. Формально не открытый вопрос
дизайн-системы — но исходный запрос пользователя, возможно, имел в виду
латиницу набора, не смену языка лейбла. Не решаю сам, фиксирую как
непереспрошенное.
