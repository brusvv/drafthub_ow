> Один файл, без номера в имени — свежесть проверяется через `agent_log.py check`,
> как у любого файла репозитория (не сравнением номеров на глаз, см. § WORKFLOW).
> Завершённая работа и история багов → `CHANGELOG.md` (не грузить лишний раз,
> там же актуальные хэши всех закрытых файлов).

## 🔑 Протокол версий — обязателен

```bash
# Перед работой
python3 scripts/agent_log.py check

# После работы
python3 scripts/agent_log.py update <агент> <задача> <файл1> [файл2 ...]
```

**Смежные задачи — можно и нужно объединять в один заход.** Если несколько
тикетов трогают один и тот же файл ради родственных целей (например
AUDIT-D1/A2/D2 — все правят `base.css` вокруг типографики/утилит/теней),
не обязательно ждать отдельного захода на каждый номер — один агент может
закрыть все разом за один `agent_log.py update`, указав составную задачу
через `+`/`и` в описании (как уже делалось для `MIGR-2+MIGR-6`,
`BASE_PATH+роутинг` и т.д.). Не объединять только если тикеты РЕАЛЬНО
независимы и трогают разные файлы без пересечений — тогда параллелить
разными агентами эффективнее, чем сериализовать в одном заходе.

## 🎯 Зона ответственности — не перечитывать весь репозиторий

Каждая задача ниже указывает **входные файлы** — это твоя зона
ответственности. Перед выполнением задачи не перечитывай весь репозиторий
и не перепроверяй каждое утверждение из этого документа. Считай, что это
результат уже выполненного архитектурного анализа. Используй его как
исходные данные и сосредоточься на реализации.

Переходи к собственному анализу только в двух случаях:
- если для выполнения рефакторинга действительно не хватает информации;
- если обнаружится явное противоречие между документом и текущим
  состоянием проекта (`agent_log.py check` — несовпадение хэша важнее
  текста задачи; текст мог устареть, хэш не врёт).

Во всех остальных случаях не трать время на повторный аудит репозитория,
а сразу выполняй описанный план. Если задача не указывает конкретные
входные файлы (общая формулировка) — сначала `agent_log.py check`, потом
только файлы из своей секции ниже, ничего больше.

**Методология обновления MD-файлов (не тратить токены на перезапись):**
- Читать только нужную секцию: `sed -n '/^## Агент LEQ/,/^## /p' AGENT_TASKS.md`
- `AGENT_FILE_LOG.md` — только через `agent_log.py update`, не руками
- Никогда не переписывать весь файл чтобы изменить 3 строки
- **Когда задача закрыта — полностью удаляется из `AGENT_TASKS.md`, не
  остаётся здесь ни в каком виде** (ни детальной секцией, ни одной строкой
  статуса). Весь текст (что сделано, почему, какие файлы) переносится в
  `CHANGELOG.md` одним блоком с обязательным форматом заголовка:
  `## <Название задачи> — <агент>, <дата ЗАКРЫТИЯ, не начала>`.
  `AGENT_TASKS.md` содержит только ОТКРЫТОЕ — если задачи нет в файле,
  значит она либо не существовала, либо уже в `CHANGELOG.md`.

## 📏 Правило размера файлов — 350 строк

Если после твоей правки файл (JS/CSS — HTML/SQL считаем отдельно, там объём
менее показателен) превышает **~350 строк**, останавливаешься и думаешь про
разбивку/рефакторинг ДО того как отдавать файл пользователю — не после. Не
обязательно делать разбивку в той же задаче (может быть не в тему), но
минимум:
- Прикинуть по каким границам файл можно расколоть (см. пример
  `sheets-bridge.js` → `sheets-auth.js`/`sheets-import.js`/`sheets-export.js`
  — разбивка по auth/import/export, а не механически пополам)

## 🔴 Открыто — приоритет

### SEC-3 — Stored XSS: `esc()`/`escAttr()` не экранируют HTML-контекст, CSV-превью вообще без экранирования
`code-verified` (17.07, Claude/LEQ) — пришло как находка внешнего security-
аудита, независимо перепроверено чтением кода и подтверждено, не принято
на слово (см. рекомендацию по порядку работы ниже — этот тикет её
реализует). Два отдельных вектора, общая причина — нет функции, которая
реально экранирует под HTML-контекст (только под JS-строку/часть атрибута):

```js
function esc(s) { return (s || '').replace(/'/g, "\\'"); }
function escAttr(s) { return (s || '').replace(/&/g, '&amp;').replace(/"/g, '&quot;'); }
```
(`render-utils.js:75,80`)

**Вектор 1 — атрибут-breakout через имя игрока.** `render-players.js:82`
(`data-name="${esc(p.name)}"`) и аналогично `render-roster.js:99,105,120,198`
— атрибут в ДВОЙНЫХ кавычках, но `esc()` не трогает `"` (только `'`).
`p.name` — свободный ввод в форме добавления/редактирования игрока
(`db-write.js`, `name` без какой-либо санитизации летит прямо в
`dbInsert('players', row)`). Имя игрока `x" onmouseover="alert(1)`
разрывает атрибут и добавляет свой обработчик — исполнится в браузере
ЛЮБОГО участника команды, открывшего вкладку Игроки. Не гипотетика,
проверено построчно чтением кода (не просто described by audit).
**Право на эксплуатацию:** любой участник с правом добавлять/редактировать
игроков (не обязательно admin) → жертва — любой, кто откроет вкладку
Игроки, включая admin/superadmin в этой же команде — кросс-пользовательский
stored XSS с эскалацией привилегий по факту жертвы.

**Вектор 2 — CSV-превью вообще без вызова esc/escAttr.**
`render-admin-import.js:153-163` (`_renderCsvPreview()`) — `${headers.join(
', ')}`, `<th>${h}</th>`, `<td>${r[h] ?? ''}</td>` — сырые ячейки CSV прямо
в `innerHTML`, без единой экранирующей функции (хуже вектора 1 — там хотя
бы неполное экранирование). Превью рендерится ДО подтверждения импорта
(`_onCsvFileSelected` → `_parseCsv` → `_renderCsvPreview`, до клика
«▶ Импортировать») — исполнится в браузере admin/superadmin в момент
открытия файла, до принятия решения импортировать или нет. Требует только
убедить admin открыть подготовленный CSV — не требует прав на запись в
команду вообще.

**Пересечение файлов:** `render-admin-import.js` также в скоупе `AUDIT-D7`
(добавляет preflight-сводку в тот же `_renderCsvPreview()`) — делать одним
агентом/заходом, не двумя параллельно, иначе конфликт правок одной функции.

**Действие:** заменить `esc()`/`escAttr()` на реальное HTML-экранирование
(`&`→`&amp;`, `<`→`&lt;`, `>`→`&gt;`, `"`→`&quot;`, `'`→`&#39;`) везде, где
результат идёт в HTML-контекст (атрибут или текстовый узел) — не только в
эти 2 конкретных места, это системная причина, значит нужен grep по всем
вызовам `esc(`/`escAttr(` (реальное число — см. `AUD-7` ниже, не гадать).
Разделить по смыслу: одна функция для HTML text/attribute context (то, что
нужно почти everywhere), отдельная — если где-то ДЕЙСТВИТЕЛЬНО нужно
экранировать под JS string literal внутри `onclick="...('...')"` (текущее
поведение `esc()` — валидно для этого узкого случая, но НЕ для контекста
HTML-атрибута одновременно, отсюда и баг: одна функция пыталась закрыть
оба контекста и не справилась ни с одним до конца). **Риск:** правка
экранирования — низкий сама по себе, но задевает много мест разом
(системная замена), тестировать не точечно. **Приоритет:** максимальный —
живая, подтверждённая уязвимость. **Агент:** первым, до всего остального
в этом файле.

### AUDIT-D7 — CSV-импорт: нет preflight-сводки перед записью

finding, теперь только здесь):**
- Файлы: превью/парсинг CSV — `render-admin-import.js` (`_renderCsvPreview`,
  вызывается из `_onCsvFileSelected` сразу после `_parseCsv`, ДО клика
  «▶ Импортировать»); сама логика импорта по типам (heroes/maps/players/
  hero_map_strength/hero_synergy/global_tiers) — `render-admin-import-data.js`
  (`_importCsv` диспетчер + `_import*` на каждый тип); резолверы —
  `sheets-import-resolve.js` (`_resolveHeroId`/`_resolveMapId`, тот же
  global scope что и Sheets-импорт).
- Сейчас `_import*` функции сами и резолвят, и пишут — нет отдельного шага
  «посчитать что получится» без записи. Для preflight-сводки нужно вынести
  resolve-шаг каждого `_import*` (маппинг/фильтрацию строк, без
  `_batchUpsert`/insert/delete) в переиспользуемую функцию, которую зовут и
  preview (только count), и сам импорт (реальная запись) — это тот самый
  принцип «не считать дважды разными способами» из описания задачи выше.
  Просто скопировать резолв-логику в `_renderCsvPreview` было бы вторым
  местом, которое разойдётся с `_import*` при следующей правке.
- «Создано» vs «обновлено» (не просто «импортировано») из формулировки
  задачи — это разные числа: текущие `_import*` делают чистый upsert, не
  различают. Чтобы посчитать честно, нужен доп. запрос существующих
  team_id-строк (или tier_entries для global_tiers) ДО импорта — сравнить
  resolved-id с уже существующими. Если это дороже чем стоит для превью —

  create/update разделения, явно об этом сказать пользователю при показе
  (не выдавать оценку за факт).
- UI: сводка встаёт в `_renderCsvPreview` (или отдельный блок над кнопкой
  `#adminImportBtn`), должна пересчитываться и при смене типа через
  `_onImportTypeChange()`, если CSV уже загружен (сейчас эта функция трогает
  только текст подсказки, не превью).

## 🎨 Design Audit — периодическая оценка UI/UX

**Когда запускать:** не после каждой мелкой задачи — раз в несколько voln
закрытых feature-задач (ориентир: после каждого MIGR-подобного блока или раз
в 10-15 закрытых пунктов), либо по прямому запросу пользователя. Разовый
"Design Research" (см. `CHANGELOG.md`, все пункты 1-10 закрыты) был такой
проверкой один раз — этот раздел формализует процесс чтобы не ждать
следующего явного запроса.

**Чеклист (что смотреть, в этом порядке):**

1. **Консистентность отступов/размеров** — сравнить `padding`/`gap`/`border-radius`
   у однотипных компонентов (все карточки между собой, все модалки между
   собой, все кнопки одного типа). Разное на глаз = не намеренное дизайн-
   решение, а дрейф — искать где именно разошлось. Заодно прогнать
   `css_dupe_check.py` по всем изменённым CSS — задвоенный простой
   селектор с расходящимися свойствами это тот же дрейф, только уже
   в коде, а не только на глаз (`python3 scripts/css_dupe_check.py
   src/css/**/*.css`, см. § WORKFLOW ниже про сам скрипт).
2. **Контраст и читаемость** — `color`/`background` пары на реальный WCAG AA
   (4.5:1 текст, 3:1 крупный) там где не проверяли явно. Особое внимание:
   `--text3`-класс (самый тихий текст) на тёмном фоне — легко случайно
   уйти ниже порога.
3. **Состояния, не только "счастливый путь"** — на каждый экран/компонент:
   что видит пользователь при (а) пустых данных (0 карточек), (б) загрузке
   (skeleton есть везде где грузится сеть?), (в) ошибке сети/RLS-отказе.
   Несогласованность здесь обычно самая заметная (где-то есть skeleton,
   где-то белый экран).
4. **Responsive-брейкпоинты** — реально открыть каждую вкладку на нескольких
   разрешениях экрана: телефон (375px ширины окна), HD/Full HD (1280–1920px,
   покрывает брейкпоинты 768/960/1200 в `responsive.css`), 2K/4K (2560px+,
   верхний брейкпоинт `responsive.css` — 1440px, дальше макет не меняется, но
   реальный масштаб/читаемость на большом мониторе всё равно стоит смотреть
   отдельно). Не только читать `@media` в коде — код может выглядеть правильным
   и всё равно ломаться на конкретной ширине из-за соседнего правила.
5. **Анимации/переходы** — либо везде есть (карточки/модалки/тосты), либо
   нигде — смешанное ощущение "тут плавно, тут дёргается" хуже чем нигде
   плавно. После POL-2/POL-3(JS) закрытия — сверить что fade-up не
   конфликтует с существующими transition на тех же элементах.
6. **Фокус-менеджмент клавиатуры** — Tab по каждой активной вьюхе, не
   застревает ли фокус в модалке, не пропадает ли видимое кольцо
   (SEC-2 даёт кольцо, но если элемент сам `tabindex="-1"` без причины —
   кольцо появляться не на чём).

**Формат отчёта** — та же таблица что в `CHANGELOG.md` Design Research:
`# | Правка | Файлы | Приоритет/риск`. Не пытаться закрыть всё одним
агентом за один заход — как и раньше, чистый CSS без пересечений уходит
одному агенту, JS-требующие пункты — координируются через таблицу выше.
**Не заводить новых пунктов "на глаз" без конкретного файла/строки** — общее
"дизайн не очень" не actionable, конкретное "modal-map.html:34 отступ 12px,
у остальных модалок 16px" — actionable.

**Каждая находка — с маркером уверенности**, не смешивать в одну кучу:
- `code-verified` — проверено чтением/grep/математикой (contrast-расчёт,
  cascade-порядок, точный `grep -c`) — можно действовать сразу.
- `needs-screenshot` — код выглядит правильным, но затрагивает реальный
  рендер (ширина панели, реальный размер шрифта на экране) — DESIGN-2
  был code-verified и оказался неверным по факту, брейкпоинт код может
  выглядеть правильно и всё равно ломаться на конкретной ширине.
- `assumed` — гипотеза, не проверялась ни чтением, ни рендером.
Не путать «я уверен» (code-verified) с «я вроде уверен» (assumed).

**Числа-находки — с датой снятия, не голым числом.** Формат:
«28 на 05.07 → 52 на 11.07» вместо «28» — счёт увеличивается по мере роста
кода, голое число из старого отчёта незаметно устаревает и вводит в
заблуждение при следующей сверке (см. `AUDIT-B5` про тот же класс проблемы
с логами файлов).

**Скриншот от пользователя = повод пересмотреть экран целиком**, не только
конкретную жалобу на нём. Практика показывает: рядом с тем что пользователь
заметил обычно находится смежное (CSP-404 рядом с жалобой на маленькие
иконки, третий дубль `.ban-chip` рядом с жалобой на отступ, реальная ширина
панели рядом с жалобой на шрифт) — читать код вокруг всего видимого на
скриншоте, не только строку с багом.

**После любой bulk-правки `style=` → `class=`** — обязательный `grep` на
задвоенные атрибуты (`class="X" class="Y"` на одном элементе) ДО того как
отдавать файл, не после. Это нашлось один раз по везению, не полагаться
что повезёт снова:
```bash
grep -n 'class="[^"]*"[^>]*class="' <файл>
```

**Статус на 16.07 (пункт 3, empty-state — уточнено, было «не проверялось
отдельно», теперь конкретно):** пункты 1-2 (Design Audit #1), 4-6 (Design
Audit #2) пройдены — детали в `CHANGELOG.md`. Пункт 3 — skeleton/error есть

### Отдельные план-файлы — правило слияния

Если аудит/анализ выдаётся отдельным файлом (`*_PLAN.md`, `*_REPORT.md` и
т.п.) вместо прямой правки `AGENT_TASKS.md` — один из двух вариантов
ОБЯЗАТЕЛЕН в тот же заход, не «на потом»:
1. Слить содержимое в `AGENT_TASKS.md` (пункты как обычно, `### PREFIX-N`);
2. Если слияние прямо сейчас нецелесообразно (например план ещё не
   провалидирован) — добавить одну строку в `AGENT_TASKS.md`:
   `⏳ ЖДЁТ МЕРДЖА: <имя файла> — <дата>, <кто выдал>`.
Отдельный файл, не упомянутый в `AGENT_TASKS.md` ни единой строкой,
считается потерянным — эта информация не переживёт следующую сессию.

### AGENT_TASKS.md и CHANGELOG.md — оба без номера в имени

Та же логика что и для кода: свежесть проверяется хэшем через `agent_log.py check`,
не сравнением номеров файлов на глаз. Никаких `_12.md`/`_13.md` не создавать —
редактировать точечно (`sed`/целевые правки) единственные файлы.
Когда задача в `AGENT_TASKS.md` закрывается — **удаляется отсюда полностью**,
весь текст переносится в `CHANGELOG.md` одним блоком с заголовком
`## <Название> — <агент>, <дата закрытия>`. Ничего не остаётся здесь даже
одной строкой — см. § «Зона ответственности» вверху файла.

**Для CSS/визуальных задач — два разных состояния, не путать:**
- «Инфраструктура готова» — переменная/класс объявлены в CSS, код существует.
- «Применено» — реально проставлено на всех местах, которые задача называла
  (проверяется тем же `grep`, которым находили исходную проблему — то же

Помечать задачу закрытой (и переносить в `CHANGELOG.md`) можно только когда
ВЕРНО ВТОРОЕ. Было дважды ложное закрытие по этой причине (POL-3 — CSS

не переведены) — если инфраструктура готова, а применение ещё нет, статус
`🔶 частично` с явным «что сделано / что осталось» (см. `FILESPLIT-1` как
пример формата частичного статуса), не `✅`.
